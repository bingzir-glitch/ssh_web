### 事项：CoPaw 对话上下文/记忆压缩机制（省 token / 防失忆）梳理

- 基本机制：每次调用模型时都会重新组装并提交上下文（不是只提交用户当轮输入）。
  - 固定注入 system prompt：由工作目录中的多个 Markdown 配置文件拼接而成（如 `SOUL.md / PROFILE.md / AGENTS.md / IDENTITY.md` 等），这部分通常“每轮都计入 token”。
  - 提交近期对话历史：在未超出上下文窗口时会携带多轮消息。

- 自动压缩（Memory Compaction）：当估算 token 接近阈值时，会触发 `MemoryCompactionHook` 对旧消息做压缩摘要。
  - 默认尽量保留最近 `3` 条消息原文不压缩（常量 `MEMORY_COMPACT_KEEP_RECENT=3`）。
  - 触发阈值与比例：
    - `max_input_length` 默认 131072 tokens（可配置）。
    - `memory_compact_ratio` 默认 0.75（达到阈值时压缩老内容的比例）。
    - `memory_reserve_ratio` 默认 0.1（压缩时预留比例）。
  - 旧消息会被汇总进压缩摘要（compressed summary），后续只带摘要 + 最近窗口，从而节省 token 并降低“长对话失忆”。

- 可用命令（用于现场确认/手动干预）：
  - `/history`：查看当前 memory 中尚未压缩的消息列表/概况。
  - `/compact_str`：查看当前压缩摘要文本。
  - `/clear`：清空历史与摘要（立刻省 token，相当于新会话）。

- 省 token 的实践建议：优先精简 system prompt（上述 md 文件内容越长，每轮固定成本越高）。
